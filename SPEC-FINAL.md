## 1. 产品概要

**一句话描述**：一个 macOS 菜单栏像素猫应用，当 Terminal.app Dock 图标出现红点（badge）时，像素猫从菜单栏掉落到屏幕上，用漫画气泡提醒你去查看终端。

**核心场景**：用户在使用 Claude Code / Codex CLI 时，终端等待命令确认或任务中断，但用户切到了其他窗口没注意到。

## 3. 我的技术推荐（需你确认）

### 3.1 开发框架：Swift + AppKit（主） + SwiftUI（设置窗口）

**我的回答**：OK，开发框架就Swift + AppKit（主） + SwiftUI（设置窗口）

**理由**：
- 菜单栏应用（NSStatusItem）、透明悬浮窗口（NSWindow）、Dock badge 检测（Accessibility API）都是 AppKit 领域，SwiftUI 在这些场景下限制多且坑多
- 设置窗口用 SwiftUI 写更快更现代，可以通过 NSHostingController 嵌入
- macOS 13+ 完全支持
- 原生轻量，无 Electron 那种 200MB+ 的体积问题

### 3.2 动画实现：Sprite Sheet + Core Animation

**我的回答**：OK，动画实现就Sprite Sheet + Core Animation实现。

**理由**：
- 像素风天然适合序列帧（Sprite Sheet），每个动作状态一组帧
- 掉落/弹跳/跳回的物理运动用 Core Animation（CAKeyframeAnimation）控制位移曲线
- 比 SpriteKit 轻量，不需要引入游戏引擎
- 比 Lottie 更适合像素风，Lottie 偏向矢量/设计师风格

**需要准备的 Sprite Sheet**：

| 动作 | 帧数（建议） | 说明 |
|------|-------------|------|
| idle | 1 帧 | 菜单栏静止头像（18×18px） |
| drop | 4-6 帧 | 掉落中的姿势 |
| land | 3-4 帧 | 落地弹跳 |
| talk | 2-4 帧 | 说话时的微动（嘴巴开合等） |
| jump_back | 4-6 帧 | 跳回菜单栏 |

第一版可以极简：idle 1 帧 + 其余用 Core Animation 位移模拟，后续再加帧动画丰富度。

### 3.3 Dock Badge 检测方案

这是技术上最关键的一环。macOS 没有公开 API 直接监听其他应用的 Dock badge 变化。可行方案：

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **A. Accessibility API** | 通过 AXUIElement 读取 Dock 进程中 Terminal 图标的 badge 属性 | 最可靠，但需要辅助功能权限 |
| **B. NSDistributedNotificationCenter** | 监听系统通知中 badge 变化相关事件 | 不一定所有 badge 变化都有通知，不够稳定 |
| **C. 定时轮询 CGWindowList** | 截取 Dock 区域检查 badge 像素 | hack 方式，不推荐 |

**推荐方案 A**，你已接受"功能优先、权限多也接受"。应用启动时引导用户授予辅助功能权限即可。

**我的回答**：方案A。

## 4. 提醒话语分类体系（需你确认/补充）

根据你提到的 Claude Code / Codex CLI 场景，我建议以下分类：

### 分类及示例（中文版，每类 5 句）

**类别 1：命令等待确认**
> 检测到 badge 出现，通用场景

1. 喵~ 终端在叫你！快去看看吧
2. 有消息啦！终端那边需要你
3. 嘿！终端亮红灯了，去瞧瞧？
4. 终端有动静了，别忘了哦~
5. 喵呜！你的终端在等你回复呢

**类别 2：长时间未响应**
> badge 持续存在超过一定时间（如 2 分钟）

1. 喂喂喂，终端等你好久了！
2. 终端都急了...快去看看吧！
3. 你是不是忘了终端还开着？
4. 已经过了好一会儿了，终端还在等你...
5. 再不去看，终端要生气了喵！

> **问题**：你是否需要更多分类？比如区分"任务完成"vs"任务中断"vs"等待确认"？如果要区分，仅靠 Dock badge 无法判断具体是哪种事件——badge 只告诉我们"有通知"，不告诉我们通知内容。要做细分需要额外手段（如读取终端窗口文本，需要屏幕录制权限）。

**我的回答**：暂时不需要更多分类，这个后续可以添加。
